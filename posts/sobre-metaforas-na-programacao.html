Sobre metáforas na programação

05 de Maio, 2020

Tenho pensado sobre a dificuldade na comunicação entre tecnologia e negócios. Não somente de código vivem os bons programadores, para se desenvolver uma solução efetiva, é essencial fazer a tradução do "tecniquês" para o "negociês" e vice-versa. É preciso garantir que a área de tecnologia esteja indo no caminho correto e é impossível fazer isso sem se tornar um bom tradutor.

Dentre as diversas ferramentas que podem ajudar nesse trabalho, a que mais tenho usado é a explicação através de analogias.

O pensamento metafórico

A principal forma do nosso cérebro absorver uma nova ideia é através da associação com ideias que já existiam previamente. Nós somos máquinas fantásticas de classificação e categorização de coisas. Constantemente estamos tentando comparar algo novo com alguma ideia que já conhecemos.

Nossa cabeça funciona igual aquele brinquedo de colocar as formas geométricas na caixa, sabe? Pra cada ideia que somos expostos, tentamos encaixar em um "buraco mental" que já existe.

É impossível criar uma nova ideia sem nenhuma base. Tente pensar em uma musica que tenha somente instrumentos musicais que você nunca ouviu antes.

A partir disso, entendemos que estamos sempre tentando associar uma nova ideia que somos expostos a algo que é presente no nosso dia a dia. Isso é o pensamento metafórico.

"Uma API é algo parecido com uma tomada"

Essa é uma frase que sempre uso quando preciso explicar uma API (Application Programming Interface, tradução Interface de Programação de Aplicação). Uma API é um conceito bastante complexo e contraintuitivo se você não possui a base técnica para entender. Não é um site, não é um aplicativo, não é visual. Você não tem onde clicar ou tocar em uma API. Segundo a wikipedia: uma API é uma interface de computação que define interações entre vários intermediários de software. Define os tipos de chamadas ou solicitações que podem ser feitas, como fazê-las, os formatos de dados que devem ser usados, as convenções a serem seguidas, etc.. Ela também pode fornecer mecanismos de extensão para que os usuários possam estender a funcionalidade existente de várias maneiras e em vários graus. (traduzido do inglês, API Wiki)

O problema com essa definição é que é necessário entender o que seriam as interações, os intermediários de software ou o que significa estender uma funcionalidade em vários graus e essas são ideias que normalmente não foram apresentadas para pessoas que trabalham no âmbito do negócio ou do produto. Eu até acho uma boa definição de API, mas é porque eu sei o que é, pra que serve e em qual contexto elas existem.

É aí que vem a metáfora da tomada: Vamos supor que você precisa de energia para ligar uma cafeteira. A entrada da tomada tem uma forma específica (padrão de três pinos?) que precisa ser compatível com a tomada da cafeteira e as tensões devem ser as mesmas (220V ou 110V). Ela também precisa estar conectada à fiação que possui energia elétrica para funcionar. Você precisa pagar por essa energia. Se você sobrecarregar essa tomada ligando um chuveiro, um secador de cabelo e uma cafeteira ela não vai ser capaz de te entregar toda a energia, a não ser que tenha sido preparada para isso. Todas essas analogias valem de alguma forma para uma API (ao menos para uma requisição do tipo GET) e trazem o entendimento de algo complexo para ideias que existem no nosso dia-a-dia.

O perigo: saltos lógicos

Quem está aprendendo esse novo conceito pode assumir características à coisa que você está explicando que na verdade não existem. Seguindo no exemplo da tomada, qualquer pessoa que quiser ligar uma cafeteira na tomada pode fazê-lo sem nenhum tipo de senha, autenticação, etc.. Sendo assim, a pessoa passa a assumir que uma API não é segura ou que obrigatoriamente qualquer pessoa pode acessa-la, o que não necessariamente é verdade, a maior parte das APIs possui um sistema de autenticação que garante a segurança da sua utilização.

seria esse o protocolo de autenticação das tomadas?

Por isso, mais do que usar boas metáforas, o importante é explicar em quais casos essa analogia não se aplica. É necessário ter cuidado na utilização em excesso dessa ferramenta porque pode causar um problema ainda maior, que é: a sensação para não-programadores de que a tecnologia é mais simples do que de fato é. E todos nós sabemos que você não quer isso, né?

...últimas considerações

Sempre é importante lembrar que a comunicação através de analogias não é uma ferramenta que funciona em todos os casos! Pra quem só tem martelo, todo problema é prego. Teste, aplique e também busque outras formas de "traduzir" o seu trabalho.
